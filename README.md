# 16-bit Pipelined ALU with Integrated Clock Gating (ICG)

## Overview

This project implements a 16-bit Arithmetic Logic Unit (ALU) designed in Logisim-evolution 4.0. The design supports 9 operations across arithmetic and logic domains, and is structured as a pipelined architecture with Integrated Clock Gating (ICG) at each pipeline stage to minimise dynamic power consumption.

The project is built from the ground up as a hierarchy of sub-circuits, from basic building blocks (D latch, adder chain, logic multiplexer) up to a fully pipelined top-level system with automatic clock enable generation.

## Architecture Overview

The design is composed of 10 sub-circuits organised in a strict hierarchy:

```
ALU_ICG_Pipelined          ← Top-level simulation circuit
│
├── ALU_with_ICG            ← ALU core with ICG integration
│   ├── ALU                 ← Main 16-bit ALU (combinational)
│   │   ├── AU              ← Arithmetic Unit (16-bit ripple adder)
│   │   └── LU              ← Logic Unit (8-input MUX)
│   ├── ICG (×4)            ← Integrated Clock Gating cells
│   │   └── DLatch          ← Negative-level-sensitive D latch
│   └── Pipelining_register (×5) ← 16-bit D flip-flop banks
│
└── Binary_count            ← 2-bit JK counter (enable sequencer)
    └── Binary_decoder      ← 2-to-4 one-hot decoder
```

## Sub-Circuit Reference

### `ALU_ICG_Pipelined` — Top Level

The top-level simulation circuit. Connects all sub-systems. The internal clock is generated by a Logisim Clock component and drives both the pipeline registers and the Binary_count enable sequencer.

| Pin | Direction | Width | Description |
|-----|-----------|-------|-------------|
| `in0` | Input | 16-bit | Operand A |
| `in1` | Input | 16-bit | Operand B |
| `OP_CODE` | Input | 6-bit | Operation selector |
| `ALU_out` | Output | 16-bit | Pipeline registered result |
| `flag_NZCV` | Output | 4-bit | Status flags: N, Z, C, V |

> **Note:** The clock and enable signals are generated internally. This circuit contains flip-flops and is sequential — Logisim's test vector engine cannot be used on this circuit directly.

---

### `ALU_with_ICG` — ALU Core with Clock Gating

Instantiates the combinational ALU, four ICG cells, and five pipelining registers. The ICG cells gate the clock to each pipeline register based on whether the register's input data has changed from its current stored value.

| Pin | Direction | Width | Description |
|-----|-----------|-------|-------------|
| `A` | Input | 16-bit | Operand A |
| `B` | Input | 16-bit | Operand B |
| `OP_CODE` | Input | 6-bit | Operation selector |
| `en0`–`en3` | Input | 1-bit each | ICG enable signals (one per pipeline stage group) |
| `clk` | Input | 1-bit | Clock input |
| `ALU_out` | Output | 16-bit | Registered ALU result |
| `flag_NZCV` | Output | 4-bit | Registered status flags |

---

### `ALU` — Combinational ALU

The core arithmetic and logic circuit. Fully combinational — no clock, no registers. Takes two 16-bit operands and a 6-bit opcode, and produces a 16-bit result with 4 status flag bits. Also exposes XOR bank debug outputs used by the ICG enable logic.

| Pin | Direction | Width | Description |
|-----|-----------|-------|-------------|
| `in0` | Input | 16-bit | Operand A |
| `in1` | Input | 16-bit | Operand B |
| `opcode` | Input | 6-bit | Operation selector |
| `en0`–`en3` | Input | 1-bit each | Passed through to ICG cells |
| `Data_out` | Output | 16-bit | ALU result |
| `flag` | Output | 4-bit | Status flags N, Z, C, V |
| `xor0`–`xor7` | Output | 16-bit each | ICG XOR bank debug outputs |

> **Test vector target:** The `ALU` circuit is the correct target for Logisim-evolution test vectors, as it is the only fully combinational circuit in the design.

---

### `AU` — Arithmetic Unit

A 16-bit ripple-carry adder constructed from 16 individual 1-bit adder cells chained in sequence. Carry propagates from bit 0 (LSB) through to bit 15 (MSB). The carry-in pin (`C_in`) allows both ADD and SUB operations — SUB is implemented by inverting operand B externally (via `SEL_B` in the opcode) and asserting `C_in = 1`, completing the two's complement subtraction.

| Pin | Direction | Width | Description |
|-----|-----------|-------|-------------|
| `in0` | Input | 16-bit | Operand A (conditioned) |
| `in1` | Input | 16-bit | Operand B (conditioned) |
| `C_in` | Input | 1-bit | Carry input (0 for ADD, 1 for SUB) |
| `Output` | Output | 16-bit | Arithmetic result |
| `C` | Output | 1-bit | Carry out (bit 16 overflow) |
| `V` | Output | 1-bit | Signed overflow flag |

---

### `LU` — Logic Unit

An 8-input multiplexer implementing all logic operations. The 3-bit `sel` input selects which operation's result is passed to the output. Inputs `in0` and `in1` may be pre-inverted by the `SEL_A`/`SEL_B` bits of the opcode before entering the LU, enabling NOT A and NOT B operations without dedicated NOT logic.

| Pin | Direction | Width | Description |
|-----|-----------|-------|-------------|
| `in0` | Input | 16-bit | Operand A (conditioned) |
| `in1` | Input | 16-bit | Operand B (conditioned) |
| `sel` | Input | 3-bit | Operation selector (b3b2b1 of opcode) |
| `out1` | Output | 16-bit | Logic result |

MUX selector mapping:

| sel | Operation |
|-----|-----------|
| 000 | Pass `in0` (NOT A when `SEL_A=1`) |
| 001 | Pass `in1` (NOT B when `SEL_B=1`) |
| 010 | A AND B |
| 011 | A NAND B |
| 100 | A OR B |
| 101 | A NOR B |
| 110 | A XOR B |
| 111 | Reserved |

---

### `ICG` — Integrated Clock Gating Cell

Implements glitch-free clock gating. The enable signal is generated by a XOR bank comparing the new ALU result against the current value stored in the downstream pipeline register. If all bits match (no change), the enable is low and the clock is suppressed, preventing unnecessary switching activity and reducing dynamic power.

The XOR comparison feeds a 16-input OR gate to produce a single-bit change-detect signal. This is ANDed with an external enable, then latched through a negative-level-sensitive D latch before passing to the final AND gate with the clock. The latch ensures the enable is stable before the rising clock edge, eliminating glitches.

| Pin | Direction | Width | Description |
|-----|-----------|-------|-------------|
| `xor0` | Input | 16-bit | New ALU result |
| `xor1` | Input | 16-bit | Current register Q output |
| `en` | Input | 1-bit | External enable from decoder |
| `clk` | Input | 1-bit | Free-running clock |
| `ICG_out` | Output | 1-bit | Gated clock to pipeline register |

---

### `DLatch` — Negative-Level-Sensitive D Latch

A transparent latch built from cross-coupled NOR gates with AND-based input steering. Transparent (output follows input) when `en = 0` (clock low phase). Holds its output when `en = 1` (clock high phase). Used inside the ICG cell to sample the enable signal during the low clock phase, ensuring it is stable before the rising edge reaches the downstream flip-flops.

| Pin | Direction | Width | Description |
|-----|-----------|-------|-------------|
| `D` | Input | 1-bit | Data input |
| `en` | Input | 1-bit | Clock enable (active low transparent) |
| `Q` | Output | 1-bit | Latched output |

---

### `Pipelining_register` — 16-bit Pipeline Register

A bank of 16 positive-edge-triggered D flip-flops sharing a common clock. Used at each stage boundary in the pipeline. The clock input is fed from an ICG cell rather than the free-running clock, so the register only latches a new value when the ICG determines the data has changed.

| Pin | Direction | Width | Description |
|-----|-----------|-------|-------------|
| `FF_in` | Input | 16-bit | Data input |
| `clk` | Input | 1-bit | Gated clock from ICG |
| `FF_out` | Output | 16-bit | Registered data output |

---

### `Binary_count` — 2-bit Binary Counter

A 2-bit ripple counter built from two JK flip-flops. Counts 00 → 01 → 10 → 11 → 00 on each rising clock edge. Its two output bits feed the `Binary_decoder` to produce a rotating one-hot enable sequence across the four ICG cells, ensuring each pipeline stage group is enabled in turn.

| Pin | Direction | Width | Description |
|-----|-----------|-------|-------------|
| `clk` | Input | 1-bit | Clock |
| `out0` | Output | 1-bit | Bit 0 (LSB) of count |
| `out1` | Output | 1-bit | Bit 1 (MSB) of count |

---

### `Binary_decoder` — 2-to-4 One-Hot Decoder

Decodes the 2-bit counter output into four mutually exclusive enable signals. At any given clock cycle, exactly one of `en0`–`en3` is high, enabling the corresponding ICG cell. This ensures that only one pipeline stage is clocked per cycle, distributing switching activity across time and further reducing instantaneous power draw.

| Pin | Direction | Width | Description |
|-----|-----------|-------|-------------|
| `in0` | Input | 1-bit | Counter bit 0 |
| `in1` | Input | 1-bit | Counter bit 1 |
| `out0` | Output | 1-bit | Enable for ICG stage 0 |
| `out1` | Output | 1-bit | Enable for ICG stage 1 |
| `out2` | Output | 1-bit | Enable for ICG stage 2 |
| `out3` | Output | 1-bit | Enable for ICG stage 3 |

Counter to enable mapping:

| count (out1, out0) | Active enable |
|--------------------|---------------|
| 00 | en0 |
| 01 | en1 |
| 10 | en2 |
| 11 | en3 |

## Opcode Format

The 6-bit opcode controls all aspects of the ALU operation:

```
  b5   b4   b3   b2   b1   b0
  |    |    |    |    |    |
  |    |    |    |    |    └── 0 = Arithmetic unit,  1 = Logic unit
  |    |    |    |    └─────── Logic MUX selector bit 0
  |    |    |    └──────────── Logic MUX selector bit 1
  |    |    └───────────────── Logic MUX selector bit 2 / Arithmetic: 0=ADD, 1=SUB
  |    └────────────────────── SEL_B: 0 = B,  1 = NOT B  (pre-inverts B)
  └─────────────────────────── SEL_A: 0 = A,  1 = NOT A  (pre-inverts A)
```

### Supported Operations

| Operation | Opcode (binary) | Description |
|-----------|----------------|-------------|
| ADD | `000000` | A + B |
| SUB | `011000` | A − B |
| AND | `000101` | A AND B |
| NAND | `000111` | A NAND B |
| OR | `001001` | A OR B |
| NOR | `001011` | A NOR B |
| XOR | `001101` | A XOR B |
| NOT A | `100001` | Bitwise NOT of A |
| NOT B | `010011` | Bitwise NOT of B |

## Status Flags

The 4-bit `flag` / `flag_NZCV` output follows ARM-convention flag ordering:

| Bit | Flag | Meaning |
|-----|------|---------|
| 3 | N | Negative — result bit 15 is 1 |
| 2 | Z | Zero — result is 0x0000 |
| 1 | C | Carry — unsigned overflow out of bit 15 |
| 0 | V | Overflow — signed overflow (sign of result contradicts signs of operands) |

## Clock Gating: How It Works

Dynamic power in a digital circuit is proportional to switching activity. Every time a flip-flop latches a new value — even if that value is identical to what it already holds — it dissipates power through capacitive switching.

The ICG prevents this by comparing the incoming ALU result against the register's current output using a 16-bit XOR bank followed by a 16-input OR reduction. If the result is the same as what is already stored, the enable is 0 and the clock is suppressed. The register does not switch, and no power is consumed for that cycle.

The D latch inside each ICG cell is critical to correct operation. Without it, any glitch on the combinational enable path could pass through the AND gate as a spurious clock pulse, causing the register to latch incorrect data. The latch is transparent during the clock's low phase, allowing the enable to settle. When the clock goes high, the latch closes and holds the enable stable, making it impossible for a glitch to reach the AND gate output during the active clock edge.

```
enable ──► [D latch, transparent when CLK=0] ──► [AND gate] ──► gated_clk
                              ▲                        ▲
                            CLK (inverted)           CLK
```

## Pipeline Structure

The ALU_with_ICG circuit contains five `Pipelining_register` stages (Reg1–Reg5). Data propagates through these stages under control of the ICG cells. The `Binary_count` → `Binary_decoder` chain generates a rotating one-hot enable that activates each stage in sequence, ensuring no two stages switch simultaneously.

A consequence of pipelining is **output latency**: a result applied at the inputs of `ALU_ICG_Pipelined` will appear at `ALU_out` only after the pipeline has been clocked through all relevant stages. For verification purposes, the combinational `ALU` sub-circuit should be targeted directly.

## Simulation Notes

### Running the Simulation

1. Open `logism_Simulation.circ` in Logisim-evolution 4.0 or later.
2. The default top-level circuit is `ALU_ICG_Pipelined`.
3. Use `Simulate → Enable Clock` to start the internal clock.
4. Set `in0`, `in1`, and `OP_CODE` via the input pins.
5. Read `ALU_out` and `flag_NZCV` after the pipeline has propagated.

### Test Vectors

Logisim-evolution's test vector engine only supports **combinational circuits**. The correct circuit to target for automated testing is `ALU`, which is fully combinational.

The test vector header for the `ALU` circuit is:

```
in0[16] in1[16] opcode[6] en0 en1 en2 en3 Data_out[16] flag[4]
```

Set `en0`–`en3` all to `1` in every test row to ensure all pipeline stage enables are active.

## Design Decisions

**Why a ripple-carry adder in the AU?** The 16 individual 1-bit adders are chained so carry propagates bit by bit from LSB to MSB. This is simple to construct and verify in Logisim, and at this scale (16 bits) the propagation delay is acceptable.

**Why a latch rather than a flip-flop in the ICG?** A flip-flop only samples its input at one instant (the clock edge). If a glitch happens to arrive at that precise moment, it would be captured. A level-sensitive latch holds its output stable for the entire high clock phase, making it physically impossible for any glitch arriving after the latch closes to propagate to the gated clock output.

**Why a one-hot enable from the Binary_count/decoder?** Activating all four ICG enables simultaneously would allow all pipeline stages to switch in the same cycle, eliminating the power spreading benefit. The rotating one-hot sequence ensures only one stage switches per clock cycle, distributing the switching activity over four cycles.

**Why does SUB share the opcode structure with ADD?** The SUB operation is implemented by setting `SEL_B = 1` (invert B) and `b3 = 1` (assert carry-in = 1) in the arithmetic unit. This is the standard two's complement subtraction identity: A − B = A + (NOT B) + 1. No separate subtractor hardware is needed.
